## 1. Project Overview

Analysr is an AI-powered universal dashboard generator. A user types a single prompt — a company name, a startup idea, a market question — and the system generates a complete, multi-tab, data-rich analytical dashboard with real data fetched from the web, AI-synthesized business intelligence, interactive charts, tables, news feeds, and a persistent AI chat assistant.

**The core innovation:** A single JSON payload generated by AI controls everything — the page title, the brand color, the tabs, the modules, the data inside each module, and the chat context. The frontend is a pure renderer. It receives JSON and draws a dashboard. It never makes decisions about what to show or how to structure information — that is entirely the AI's job.

**What it is not:** A BI tool that connects to your database. A chatbot. A report generator. It is a zero-setup intelligence dashboard that appears from a single sentence.

## 2. The Core Idea

### The Problem It Solves

Currently, getting a comprehensive business analysis requires:
- Hours of research across multiple sources
- Manual chart creation in Excel or Tableau
- Jumping between tabs, documents, and tools
- Expertise in data visualization to make it readable

Type: `"Analyse Zomato's business model"` → Get a full dashboard in ~7 seconds with:
- Revenue trend charts with real financial data
- Market size and TAM/SAM visualization
- Competitor comparison tables
- Live news feed with sentiment badges
- Business Model Canvas
- Risk assessment module
- KPI cards with sparklines
- A persistent AI analyst explaining everything in the chat panel

## 3. UI & Layout Architecture

### Screen Structure

The screen has three distinct zones that never overlap:

```
┌─────────────────────────────────────────────────────────────┐
│  TOPBAR — Logo | Tabs | Icons | Avatar                      │  52px fixed
├──────────────┬──────────────────────────────────────────────┤
│              │                                              │
│  CHAT PANEL  │         MODULE GRID                          │
│  (fixed      │         4 columns × 4 rows                   │
│   210px)     │         16 cells available                   │
│              │         Each cell = equal fraction of        │
│              │         remaining width & height             │
│              │                                              │
└──────────────┴──────────────────────────────────────────────┘
```

### Topbar

The topbar is fixed at 52px height. It contains three zones:

**Left zone — Brand identity (dynamic, AI-controlled):**
- Square logo mark (34×34px, 9px border radius) in brand color showing company initials
- Company name in 14px/600 weight with −0.3px letter spacing
- Subtitle "Business analysis" or "Startup evaluation" in 10px muted

**Center zone — Tabs (dynamic, AI-controlled):**
- Horizontal pill tabs, AI decides count (3–6) and labels
- Active tab: white background, 1px border, 500 weight
- Inactive tabs: ghost style, hover shows light background
- Active tab has a 4px brand-colored dot indicator

**Right zone — Controls (always fixed):**
- Search icon button (30×30px, rounded square)
- Notification bell with red dot badge
- Settings gear icon
- User avatar circle with initials

### Chat Panel

The chat panel is a fixed-width column (210px) on the left side of the body. It is **not a module**. It does not occupy grid cells. It is a separate UI region with its own layout.

It contains:
- Header with green online pulse dot + "AI analyst" label
- Scrollable message thread (AI bubbles left-aligned, user bubbles right-aligned, brand red)
- Text input with send button at the bottom, always visible

The chat panel never moves, never closes, never minimizes. It is always co-visible with the data modules. This is intentional — the user reads the AI explanation on the left and looks at the chart it references on the right without any context switching.

### Module Grid

The module grid occupies the remaining width after the chat panel. It is a CSS grid of 4 columns × 4 rows with 8px gaps between cells.

```
4 columns × 4 rows = 16 cells maximum
Each cell = (remaining_width - 3 × 8px gaps) / 4 wide
Each cell = (body_height - 3 × 8px gaps) / 4 tall
```

Modules span multiple cells. A 3×3 module occupies 9 cells. A 1×1 module occupies 1 cell. The AI specifies the span for each module in its JSON output.

### Tabs

Tabs are rendered in the topbar. Switching tabs swaps which modules are displayed in the module grid. The chat panel does not change when tabs switch — the AI chat has context for the entire dashboard across all tabs.

**Critical rule:** Tabs are not modules. They are not in the grid. They do not consume grid space. The grid is always 4×4 regardless of how many tabs exist.

- Has a white background (`#FFFFFF`)
- Has a 1px border (`#EBEBEA`)
- Has a 12px border radius
- Has a 3px colored top accent stripe (the only color that varies)
- Has a title and subtitle in the content area
- Contains one or more visual elements (charts, tables, numbers, lists)
- Is generated as HTML by an AI call

## 5. The Two-State Module System

Every single module exists in exactly two states. This is one of the most important design decisions in the system.

### State A — Compact (visual only)

When a module is at its smaller size, it shows **only the visual** with zero text:
- A bar chart shows bare bars with no labels, no axes, no title, no legend
- A gauge shows just the arc fill with no percentage text
- A donut shows just the ring segments with no legend
- A KPI card shows just a large number, nothing else
- A news feed shows just one headline, no source or date
- A table shows a color-density heatmap of cells, no text content

**Rule:** In compact state, the user reads the *shape* and *pattern* of the data. Not the data itself. A rising bar chart communicates "this is growing" without any numbers being visible.

### State B — Expanded (full detail)

When a module is at its larger size, everything appears:
- Full axis labels with values
- Chart title and subtitle
- Legend with color keys
- Footer stats row (e.g., "2024: $23.2B · YoY: +8.1% · 5yr CAGR: 4.2%")
- Tooltip on hover
- Table shows all rows with column headers and sort controls
- News shows thumbnail + full headline + source + date + sentiment badge

### How the Switch Works

The module component does NOT switch between two separate React components. It is ONE component that uses CSS container queries:

```css
.module { container-type: size; container-name: mod; }

/* Compact — visual only */
@container mod (max-width: 200px) {
  .mod-title, .mod-subtitle, .axis-labels,
  .legend, .footer-stats, .source-text { display: none; }
  .visual-area { height: 100%; padding: 6px; }
}

/* Standard — title + visual */
@container mod (min-width: 201px) and (max-width: 340px) {
  .mod-title { display: block; }
  .legend, .footer-stats { display: none; }
}

/* Full — everything */
@container mod (min-width: 341px) {
  .mod-title, .mod-subtitle, .axis-labels,
  .legend, .footer-stats { display: block; }
}
```

When `react-grid-layout` resizes a module, the CSS responds automatically. Zero JavaScript. Zero state management. Zero re-rendering. The module self-adapts purely based on how much space it has.

### AI Controls Default Size

The AI specifies two sizes per module in its JSON output:
- `default_size`: The size it starts at when the dashboard loads
- `expanded_size`: The size it snaps to when the user clicks to expand

The AI also assigns a `priority`:
- `hero` → starts at expanded_size immediately (the main focus of the tab)
- `high` → starts at default_size, prominently placed
- `medium` → starts at default_size
- `low` → starts at compact, user expands if interested

font-size: 10px; font-weight: 400; color: #9B9B97;

### Spacing Tokens

```
/* Gap between elements inside a module */
gap: 6px | 8px | 12px | 16px   (only these four values)

/* Card content padding — always exactly this */
padding: 12px 14px

/* Inner element border radius */
border-radius: 4px | 6px | 8px   (never above 8px for inner elements)

/* Card border radius — always exactly this */
border-radius: 12px

/* NEVER use margin — always use gap or padding */
```

### Card Shell — Always Identical

Every module card uses this exact shell structure:

```html
<div class="module-card">
  <!-- Accent stripe — always first child -->
  <div class="accent-bar" style="background: [accent_color]; height: 3px; width: 100%;"></div>

  <!-- Content area — always second child -->
  <div class="card-content" style="padding: 12px 14px; flex: 1; display: flex; flex-direction: column; gap: 8px; overflow: hidden;">
    <!-- Module title -->
    <div class="mod-title">Title here</div>
    <!-- Module subtitle -->
    <div class="mod-sub">Subtitle here</div>
    <!-- Visual content -->
    <!-- Footer stats -->
  </div>
</div>
```

### Chart Injection — How Charts Work Inside AI-Generated HTML

The AI **never writes chart code**. Instead, it places a placeholder div:

```html
<div
  class="chart-block"
  data-type="bar"
  data-series="[19.2, 21.1, 19.2, 23.2, 25.0, 26.1]"
  data-labels='["FY19","FY20","FY21","FY22","FY23","FY24"]'
  data-unit="$B"
  data-accent="brand"
  data-compact="false"
  style="width: 100%; height: 140px;"
></div>
```

After the HTML is mounted in the DOM, a chart injector script scans for every `.chart-block` div, reads its data attributes, and mounts an ECharts instance inside it with the global design theme. The AI had zero involvement in chart rendering. Every chart looks identical because they all use the same ECharts config.

### Hard Rules for HTML Generation AI

The validator will reject any HTML that contains:
1. Inline style colors not in the palette
2. Font sizes below 10px or above 32px
3. Any JavaScript code
4. `position: fixed` or `position: absolute`
5. `z-index` usage
6. `!important` declarations
7. External font imports or CDN links
8. `margin` (use `gap` and `padding` only)
9. Any chart library code (use `.chart-block` only)
10. `border-radius` above 12px on any element

## 7. AI HTML Generation Layer

### The Philosophy

The HTML generation AI has full creative freedom over layout and information hierarchy but zero freedom over styling. It decides:
- Where elements are positioned (flexbox / grid layout)
- What information to emphasize (big number vs small label)
- How many columns of stats to show
- Whether to show the chart prominently or as a supporting element
- How to frame the data narrative within the card

It cannot decide:
- Colors (only from the palette)
- Font sizes (only from the scale)
- Spacing (only from the tokens)
- How charts look (chart-block handles that)
- Border radius (fixed values only)

### Why This Works

A human designer making a "gross margin" card might decide: large "44.6%" number dominating the top-left, a small sparkline in the top-right showing the declining trend, three comparison stats at the bottom (5yr avg, industry benchmark, vs competitor). That's a creative layout decision driven by understanding the data. The AI can make that same decision. What it cannot do is choose `color: #E02020` instead of `#DC2626` or `padding: 15px` instead of `12px 14px`. Those are locked. The creative latitude is real. The consistency is enforced.

### The Batch Structure

Cards are generated in batches of 4. Each batch call receives:

```json
{
  "design_system": "... full rules ...",
  "neighbour_context": {
    "already_done": [
      "revenue_chart: hero 3×3 bar chart, red accent, dominant data visualization",
      "franchise_kpi: 1×1 number card, green accent, compact stat"
    ],
    "design_note": "Top section is already heavy with charts. Your 4 cards are supporting context. Lean toward text-forward layouts, narrative framing, and lighter visual weight."
  },
  "cards": [
    { "id": "...", "type": "...", "size": "...", "data": {...} },
    ...4 cards total
  ]
}
```

The `design_note` tells the AI how to contrast its batch visually against what's already rendered. This prevents monotony — not every card ends up being a large chart.

## 8. Data Layer — What Data We Collect & How

### Data Sources

| Source | Tool | What It Provides | Cost |
|--------|------|-----------------|------|
| Live news | Tavily API | Latest articles, sentiment | Free (1,000/month) |
| Financial data | yFinance | Revenue, margins, stock data | Free forever |
| Web content | Crawl4AI (self-hosted) | Any webpage as clean markdown | Free forever |
| Market research | Tavily search | TAM, industry reports | Free tier |
| AI knowledge | Gemini Flash | Historical context, estimates | Free tier |

### What Gets Collected Per Query

For a company query ("Analyse Zomato"):

```
News fetch (Tavily):
  → "Zomato latest news 2026"
  → Returns: 5 articles with title, url, snippet, date

Financial fetch (yFinance):
  → ticker: "ZOMATO.NS"
  → Returns: revenue by year, gross margin, operating margin, market cap

Competitor fetch (Crawl4AI + Tavily):
  → Search: "Zomato competitors India food delivery 2026"
  → Scrape top results for company names + key metrics

Market data (Tavily):
  → Search: "India food delivery market size 2026 TAM"
  → Returns: market size figures, growth rates

AI knowledge fill (Gemini):
  → Everything that APIs couldn't find
  → Business model details, strategic context, historical milestones
```

For a startup query ("Food delivery startup India $50k"):

```
Market data (Tavily):
  → TAM/SAM for India food delivery
  → Recent funding rounds in this space

Historical failures (Tavily + Crawl4AI):
  → "failed food delivery startups India lessons"
  → Returns: Foodpanda, TinyOwl, Ola Foods case studies

Competitor landscape (Tavily):
  → Swiggy, Zomato, Blinkit current metrics

Cost benchmarks (AI knowledge):
  → CAC estimates, delivery cost per order, tech infra costs
  → Estimated burn rates for similar startups
```

After all parallel fetches, data is merged into one bundle:

```json

      "source": "Economic Times",
      "url": "https://...",
      "date": "2026-03-15",
      "snippet": "...",
      "source_type": "live"
    }
  ],
  "financials": {
    "revenue": {"FY22": 4192, "FY23": 8635, "FY24": 12114},
    "gross_margin": {"FY22": 28.1, "FY23": 34.5, "FY24": 41.2},
    "market_cap": 185000,
    "currency": "INR_CR",
    "source_type": "live"
  },
  "market": {
    "tam": 850000,
    "sam": 220000,
    "growth_rate": 18.4,
    "currency": "INR_CR",
    "source_type": "estimated"
  },
  "competitors": [
    {"name": "Swiggy", "revenue": 11000, "status": "private"},
    {"name": "Blinkit", "revenue": 3200, "parent": "Zomato"}

  "context": "AI-synthesized narrative about Zomato's evolution...",
  "source_type": "ai_generated"

```

Every data point is tagged with `source_type`: `live` (from API), `estimated` (AI approximation), or `ai_generated` (AI knowledge only). This tells the director AI how confident to be about each number when deciding how prominently to display it.

## 10. n8n Workflow — Complete Build Guide

### What is n8n

n8n is an open-source workflow automation tool that runs locally on your laptop. It provides a visual canvas where you connect nodes — each node is one operation (make an API call, run some code, call an AI). It handles parallel execution, merging, error handling, and retries natively without any code.

For this project, n8n acts as the entire backend pipeline. The Next.js frontend sends a POST request to an n8n webhook. n8n runs the whole pipeline — classifier, data collection, director AI, HTML generation — and sends back the complete JSON. The frontend receives it and renders the dashboard.

### Installation

```bash
# Run this in terminal. That's it.
npx n8n

# n8n opens at: http://localhost:5678
# No account needed for local use
# All data stays on your machine
# Free forever for self-hosted
```

### Credentials Setup

Before building nodes, add these credentials in n8n Settings → Credentials:

1. **Google Gemini API**
   - Type: "Google Gemini API"
   - API Key: your key from aistudio.google.com (free)

2. **Tavily API**
   - Type: "HTTP Header Auth"
   - Name: "Authorization"
   - Value: "Bearer tvly-your-key-here"
   - Get key at: tavily.com (free, 1,000/month)

#### NODE 1: Webhook Trigger

**Type:** Webhook  
**HTTP Method:** POST  
**Path:** `generate`  
**Response Mode:** "When last node finishes"

This node creates the URL: `http://localhost:5678/webhook/generate`

Your Next.js frontend will POST to this URL with:

### Chart Injector

```typescript
// lib/chartInjector.ts
import * as echarts from 'echarts';
import { chartTheme } from './chartTheme';

export function injectCharts(brandColor: string) {
  const placeholders = document.querySelectorAll('.chart-block');

  placeholders.forEach((el) => {
    const type = el.getAttribute('data-type') || 'bar';
    const series = JSON.parse(el.getAttribute('data-series') || '[]');
    const labels = JSON.parse(el.getAttribute('data-labels') || '[]');
    const unit = el.getAttribute('data-unit') || '';
    const accent = el.getAttribute('data-accent') || 'brand';
    const compact = el.getAttribute('data-compact') === 'true';

    const accentMap: Record<string, string> = {
      brand: brandColor,
      green: '#16A34A',
      amber: '#D97706',
      purple: '#7C3AED',
      teal: '#0D9488',
      blue: '#2563EB',
    };

    const color = accentMap[accent] || brandColor;
    const chart = echarts.init(el as HTMLElement, chartTheme);

    chart.setOption(buildChartOption(type, series, labels, unit, color, compact));
  });
}

function buildChartOption(type, series, labels, unit, color, compact) {
  const base = {
    animation: true,
    animationDuration: 600,
    grid: compact
      ? { top: 4, right: 4, bottom: 4, left: 4 }
      : { top: 12, right: 16, bottom: 24, left: 40 },
  };

  if (type === 'bar' || type === 'sparkline') {
    return {
      ...base,
      xAxis: compact ? { show: false } : { type: 'category', data: labels, axisLine: { show: false } },
      yAxis: compact ? { show: false } : { type: 'value', axisLabel: { formatter: `{value}${unit}` } },
      series: [{
        type: 'bar',
        data: series,
        itemStyle: {
          color: (params) => {
            const opacity = 0.3 + (params.dataIndex / series.length) * 0.7;
            return color + Math.round(opacity * 255).toString(16).padStart(2, '0');
          },
          borderRadius: [3, 3, 0, 0],
        },
      }],
    };
  }

  // Add line, pie, donut, etc. following same pattern
  return base;
}
```

import asyncio

        result = await crawler.arun(url=url)
        return result.markdown  # Clean markdown, ready for AI

## 15. Build Order — What to Build First

### Phase 1: Frontend with Dummy Data (Week 1)

**Goal:** A fully working, beautiful dashboard UI that renders from a hardcoded JSON file.

1. Set up Next.js project with Tailwind
2. Build Topbar component (static, hardcoded McDonald's data)
3. Build ChatPanel component (static messages)
4. Build ModuleGrid with CSS container queries
5. Write dummy JSON payload (hand-craft 8 modules manually)
6. Build ModuleCard that injects HTML from the dummy payload
7. Build ChartInjector that mounts ECharts
8. Build LoadingOverlay and reveal animation
9. Wire the reveal animation to a button click for demo

**At end of Phase 1:** You have a fully rendered, beautiful dashboard that works perfectly — with fake data. This is what you show early for feedback.

### Phase 2: n8n Pipeline (Week 2)

**Goal:** n8n returns real JSON to replace the dummy data.

1. Install n8n locally (`npx n8n`)
2. Build and test Node 1 (webhook) → verify Next.js can talk to it
3. Build Node 2 (classifier) → verify Gemini responds correctly
4. Build Nodes 4–7 (data collection) one at a time, test each
5. Build Node 8 (merge) → verify all branches combine
6. Build Node 10 (director) → this takes time to get the prompt right
7. Build Nodes 12–15 (HTML batches) → iterate on prompt
8. Build Nodes 17–18 (validate + respond)
9. Point Next.js at the real webhook URL
10. Watch real data flow into your already-polished frontend

### Phase 3: Polish (Week 3)

1. Handle edge cases (no financial data for private companies)
2. Chat panel — wire to a simple Gemini call with dashboard context
3. Tab switching animation
4. Mobile-responsive adjustments
5. Loading state text ("Fetching live data... Building your dashboard...")
6. Error states when pipeline fails
